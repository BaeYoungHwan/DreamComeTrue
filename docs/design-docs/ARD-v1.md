## 1. 품질 속성 목표 (Quality Attributes)

| 속성 | 목표 | 측정 방법 | 우선순위 |
|------|------|-----------|----------|
| 성능 | 데이터 조회·입력 반응 속도 1초 미만 | 로컬 수동 측정 (Streamlit 페이지 반응 시간) | High |
| 가용성 | 별도 SLA 타겟 없음 (개인 사용) | - | Low |
| 보안 | 외부 노출 없는 로컬 DB, 자동 백업으로 데이터 유실 방지 | 백업 파일 생성 여부 확인 | High |
| 확장성 | 단일 사용자·소량 데이터 기준, 별도 부하 대응 불필요 | - | Low |
| 유지보수성 | 핵심 도메인(재고/판매) 로직 단위 테스트 확보 | 핵심 시나리오 테스트 통과 | Medium |

## 2. 아키텍처 제약사항

### 기술적 제약
- Python 3.12 + Streamlit + SQLite 고정, 다른 프레임워크 임의 도입 금지
- 인터넷 연결이 불안정한 농장 환경에서도 완전히 로컬로 구동 가능해야 함

### 조직적 제약
- 1인 개발이므로 운영·배포 복잡도 최소화 (CI/CD, 컨테이너 오케스트레이션 불필요)

### 예산/인프라 제약
- 외부 유료 API 의존성 전면 배제 (비용 $0 목표)

## 3. 주요 아키텍처 결정 (초안)

> 상세 결정 기록은 `docs/design-docs/adr/` 에 ADR로 작성합니다.

| 결정 영역 | 선택 | 근거 |
|-----------|------|------|
| 레이어 구조 | UI(Streamlit) / 도메인(inventory, sales) / 데이터(SQLite) 3계층 | 소규모 단일 사용자 앱에 적합한 최소 구조 |
| 인증 방식 | 없음 (단일 사용자, 외부 미노출) | MVP 제외 사항으로 명시된 회원가입/권한 관리 불필요 |
| 데이터 저장 | SQLite 파일 (data/farm.db) | 외부 DB 서버 없이 완전 로컬 구동 가능 |
| 배포 전략 | 로컬 PC 실행 우선, Streamlit Community Cloud는 선택지 | 인터넷 불안정 환경 대응 |

## 4. 리스크

| 리스크 | 영향도 | 발생 가능성 | 완화 전략 |
|--------|--------|-------------|-----------|
| SQLite 파일 손상/유실 (PC 오류, 정전 등) | High | Medium | 자동 백업(주기적 파일 복사) 기능 필수 구현 |
| 재고 음수 등 데이터 정합성 오류 | Medium | Medium | 도메인 규칙(재고 음수 불가 등)을 코드 레벨에서 검증 |
| 1인 개발로 인한 검증 공백 | Medium | Medium | 핵심 시나리오(입고→출고→정산) 단위 테스트로 보완 |
| Streamlit Community Cloud 배포 시 로컬 SQLite 파일 휘발성 | Low | Low | 배포 전 로컬 우선 운영을 기본값으로 유지, 배포는 선택적 검토 |

## 5. 검증 기준

아키텍처가 이 ARD를 만족하는지 확인하는 방법:

- [ ] 반응 속도 1초 미만 체감 확인 (로컬 실행 기준)
- [ ] 자동 백업 파일이 정상 생성되는지 확인
- [ ] 재고 음수 방지 등 핵심 도메인 규칙 테스트 통과
- [ ] 주요 결정사항 필요 시 `docs/design-docs/adr/`에 ADR 작성
